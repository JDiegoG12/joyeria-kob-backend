# Contexto del proyecto

Estoy desarrollando una tienda de joyas con backend en Express + TypeScript + Prisma, usando una arquitectura MVC con **Puertos y Adaptadores (Ports & Adapters)**. Cada feature tiene su propia estructura y las dependencias se invierten mediante interfaces. El repositorio está en GitHub: https://github.com/JDiegoG12/joyeria-kob-backend/tree/develop

## 📁 Estructura actual (relevante para la implementación)

- `prisma/schema.prisma` → esquema de la base de datos (actualmente sin modelo User)
- `prisma/seed.ts` → script de siembra (actualmente solo carga categorías, productos y precio del oro)
- `src/features/categories/` → ejemplo de referencia (controllers, facade, ports, services, dtos, routes)
- `src/features/products/` → CRUD de joyas
- `src/api/middlewares/` → middlewares transversales (auth, manejo de errores)
- `src/shared/` → utilidades y constantes compartidas
- `src/config/swagger.config.ts` → configuración de Swagger/OpenAPI

## 🧱 Reglas de arquitectura que debes respetar

1. **Controller** → solo maneja `Request`/`Response`, llama al facade, nunca a la BD.
2. **Facade** → implementa la interfaz del puerto, orquesta servicios, valida negocio.
3. **Port (interfaz)** → define el contrato que el facade debe cumplir.
4. **Service** → contiene lógica de acceso a datos (consultas Prisma).
5. **DTOs** → se usan para formatear respuestas según el rol (CLIENT / ADMIN).
6. **Rutas** → cada feature tiene su `routes.ts` con anotaciones `@openapi` para Swagger.
7. **Formato de respuesta unificado** → `{ success, data, message }` (ver README del repo).

## 🎯 Objetivo: Implementar autenticación y autorización (JWT)

Debo crear el feature **users** con las siguientes características:

### 1. Modelo `User` en Prisma
- Añadir modelo `User` con campos:
  - `id` (String, UUID, @db.Char(36))
  - `email` (String, @unique)
  - `password` (String, hasheada con bcrypt)
  - `role` (enum UserRole: `CLIENT`, `ADMIN`)
  - `createdAt`, `updatedAt` (DateTime)
- Añadir el enum `UserRole` justo antes del modelo.
- Ejecutar migración (no es necesario que la IA la genere, solo dar el comando `npx prisma migrate dev --name add_user_model`).

### 2. Endpoints de autenticación (públicos)
- **`POST /api/auth/register`** → crea un usuario con rol `CLIENT` (por defecto).
  - Recibe `email`, `password` (y opcionalmente `role`, pero ignorado si no es admin).
  - Valida email único, hashea password con bcrypt (salt rounds = 10).
  - Retorna DTO `UserResponseDTO` (sin password).
- **`POST /api/auth/login`** → valida credenciales y retorna JWT.
  - Recibe `email`, `password`.
  - Verifica con bcrypt.compare.
  - Si es correcto, genera token JWT con payload: `{ id, role }`.
  - Expiración: 1 día.
  - Retorna `{ token, user: UserResponseDTO }`.

### 3. Middlewares (en `src/api/middlewares/`)
- **`authMiddleware`**:
  - Extrae token del header `Authorization: Bearer <token>`.
  - Verifica firma con `JWT_SECRET` (de `.env`).
  - Adjunta `req.user = { id, role }`.
  - Si no hay token o es inválido → 401.
- **`requireAdmin`**:
  - Verifica que `req.user.role === 'ADMIN'`.
  - Si no → 403.
